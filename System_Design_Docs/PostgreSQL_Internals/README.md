# PostgreSQL Internal Architecture

As we discussed earlier, Postgres is a client-server architecture based database server, designed for scale and concurrency. A dedicated server process runs and manges storage, transctions, concurrency, recovery and client requests.

The core components of postgres server are

1. Parser
2. Planner (Optimizer)
3. Executor
4. Transactions Manager
5. Lock Manager
6. Buffer Manager

Every SQL query that come to the server will go though the flow

Query → Parser → Planner → Executor → Buffer Manager → Heap/Indexes → WAL → Disk

This layered architecture allows PostgreSQL to efficiently support thousands of concurrent transactions while providing strong guarantees for consistency, durability, and crash recovery. Each component has a clearly defined responsibility, enabling PostgreSQL to scale from small applications to enterprise-grade database deployments.

## Query Parser

The parser stage in postgres consists of two steps

1. Parser
2. Transfromation Process

#### Parser

The parser validates that the input SQL query string has valid syntax. If the syntax is valid, it generates a *parse tree*, otherwise it will throw an error. The parser itself consists of two parts

1. Lexer, responsible for identifiing known identifiers, key words etc. It tokenizes the entire query string and that is given to the parser.
2. Parser, consists of the grammer and actions that are executed. Based on the tokens generated by the lexer, the parser will generate the *prase tree* which will also contain all the acitons

Once the parser completes its process, the *parse tree* is handed to the transformation process.

#### Transformation Process

The transformation process takes the *parse tree* and does the semantic interpretation to understand which tables, functions, and operators are referenced by that query, the output from this step is called the *query tree.*

## Query Planner

Then planner creates an optimal execution plan from the generatd query tree. A given query can be executed in different ways, where each way gives the same output. If it is compuationally feasiable, the planner will examine these different staratigies and attempt to select the one that will execute the fastest. 

Consider this example

```jsx
SELECT * 
FROM orders 
WHERE customer_id = 100;
```

The planner may choose between multiple strategies:

1. Sequential scan
2. Index Scan

If an index exists on `customer_id`, the planner estimates whether using the index will be cheaper than scanning the entire table. For a small table, a sequential scan is often faster because reading the table sequentially avoids the overhead of index lookups. For a large table with highly selective predicates, an index scan is usually preferred.

The planner estimates the cost of each strategy using statistics collected by the `ANALYZE` command, including:

- Number of rows in the table
- Distribution of column values
- Number of distinct values
- Null fractions
- Correlation between data and physical storage

These statistics allow PostgreSQL to estimate how many rows a query is likely to return and the amount of disk I/O and CPU work required for each execution plan.

## Executor

After the planner selects the optimal execution plan, the plan is passed to the PostgreSQL executor. The executor is responsible for executing the plan tree and producing the final result set.

Unlike the planner, which determines how a query should be executed, the executor performs the actual work of reading tables, traversing indexes, applying filters, performing joins, sorting rows, and computing aggregate functions.

### Demand-Pull Execution Model

PostgreSQL executes queries using a **demand-pull** (or **iterator**) execution model. Every node in the execution plan implements a common interface that produces **one tuple (row) at a time** whenever its parent node requests it.

Instead of executing the entire plan at once, execution begins at the root of the plan tree. Whenever a node requires more input, it recursively requests another row from its child nodes. This continues until a leaf node (typically a table or index scan) retrieves a row from disk or shared memory.

## Indexes

Indexes are auxiliary data structures that allow PostgreSQL to locate rows efficiently without scanning an entire table. Instead of searching every tuple in a heap table, PostgreSQL first searches the index and then uses the result to locate the corresponding tuple in the table.

The default and most commonly used index type in PostgreSQL is the **B-Tree index**, although PostgreSQL also supports several specialized index types such as Hash, GIN, GiST, BRIN, and SP-GiST.

Unlike SQLite, PostgreSQL stores tables and indexes as separate storage structures. Instead of storing the actual row, each index entry stores:

- The indexed key.
- A **Tuple Identifier (TID)** pointing to the row's physical location in the heap.

### B-Tree Structure

A PostgreSQL B-Tree is a balanced search tree consisting of fixed-size pages (typically 8 KB).

There are three types of pages:

- **Root page** – Entry point into the tree.
- **Internal pages** – Store separator keys and pointers to child pages.
- **Leaf pages** – Store indexed keys and their corresponding TIDs.

Searching an index begins at the root page and repeatedly follows child pointers until the desired leaf page is reached.

### Index Scan

Consider the query:

```
SELECT *
FROM users
WHERE email = 'alice@example.com';
```

If an index exists on `email`, PostgreSQL performs the following steps:

Query → B-Tree Index → Locate TID → HEAP → Return

The executor first traverses the B-Tree to locate the matching key. The associated TID is then used to fetch the corresponding tuple from the heap table. Before returning the row, PostgreSQL performs an MVCC visibility check to ensure that the tuple is visible to the current transaction.

### Index Maintenance

Indexes are automatically maintained whenever rows are inserted, updated, or deleted.

- **INSERT** adds a new index entry.
- **DELETE** removes or invalidates the corresponding entry.
- **UPDATE** may remove the old index entry and create a new one if an indexed column changes.

Since every modification to indexed columns requires corresponding index updates, indexes improve read performance at the cost of slower write operations and additional storage.

### Specialized Index Types

While B-Tree indexes are suitable for most equality and range queries, PostgreSQL provides several specialized index types for different workloads:

- **Hash** – Equality lookups.
- **GIN (Generalized Inverted Index)** – Arrays, JSONB, and full-text search.
- **GiST (Generalized Search Tree)** – Geospatial data, range types, and nearest-neighbor searches.
- **BRIN (Block Range Index)** – Very large, naturally ordered tables such as time-series datasets.
- **SP-GiST** – Space-partitioned data structures such as tries and quadtrees.

The PostgreSQL planner automatically chooses the most appropriate index type based on the query, available indexes, and estimated execution cost.
